# SWG Unified Launchpad:  Architecture

One launcher for every SWG server. Server operators describe their game in a hosted `manifest.json`; the launcher gives each server its own isolated game folder with full patching, settings, and launch capability:  no per-server launcher installs.

## The three moving parts

**1. The registry** (you control it). A single `registry.json` hosted somewhere you control (GitHub repo recommended:  free hosting via raw.githubusercontent.com, full audit history). It lists every approved server and points to each server's manifest URL. The launcher downloads this on startup. Adding a server to the launcher = merging one entry into this file. Removing a bad actor = deleting one entry.

**2. Server manifests** (each server controls theirs). Every server hosts a `manifest.json` at a stable URL. It contains everything the launcher needs: login host/port, client executable, patch file locations, branding, news feed. Operators update their files and manifest whenever they patch:  no coordination with you required. This is the "submit the necessary variables" piece: the manifest IS the submission.

**3. The launcher** (C#/.NET 8, WPF). Fetches registry → fetches manifests → renders a server list. Per server it can install, verify/patch, edit settings, and launch:  replicating what each server's bespoke launcher does, generically.

## Why a hosted manifest beats baked-in server configs

Every existing SWG launcher (SWGEmu Launchpad, Legends, Restoration, Finalizer) is the same machine under a different coat of paint: compare local files against a server-published list of (path, size, hash), download what differs, write the login server address into a config file, run the client. The differences between launchers are *data*, not *code*. So we move the data out of the launcher and into a manifest the server hosts. The launcher never needs an update when a server patches.

## Server manifest schema (v1)

```json
{
  "manifestVersion": 1,
  "serverId": "sunrunner-home",
  "name": "Sunrunner's Home",
  "description": "Pre-CU server with accelerated XP and custom content.",
  "era": "Pre-CU",
  "website": "https://example.org",
  "discord": "https://discord.gg/example",
  "bannerUrl": "https://patch.example.org/banner.png",
  "newsUrl": "https://example.org/news.rss",
  "statusUrl": "https://login.example.org/status.xml",
  "login": { "host": "login.example.org", "port": 44453 },
  "client": {
    "executable": "SWGEmu.exe",
    "arguments": "",
    "allowMultiInstance": true
  },
  "files": {
    "baseUrl": "https://patch.example.org/files/",
    "fileManifestUrl": "https://patch.example.org/files.json"
  }
}
```

Field notes: `serverId` is a permanent lowercase-kebab identifier (it names the install folder, so it can never change). `era` is freeform but suggested values are `Pre-CU`, `CU`, `NGE`, `Hybrid`. `client.executable` is whatever binary the server ships (`SWGEmu.exe`, `SwgClient_r.exe`, custom). `allowMultiInstance: false` lets a server opt out of dual-boxing if their rules forbid it. `statusUrl` (optional) feeds the live player count:  the launcher parses SWGEmu-style status XML or simple JSON (`{"online": n}`). `newsUrl` (optional, RSS/Atom) feeds the card's headline. Liveness itself (the green/red dot) is a TCP connect to `login.host:port`, refreshed every 60 seconds.

User-added servers: beyond the moderated registry, players can paste any manifest URL via "Add server". These render with a "custom" badge and a removal control, persist in the player's local settings, and follow the identical install pipeline:  same validation, same hash verification, same sandboxed folder. A registry-listed and a custom server can never collide because `serverId` is namespaced by folder. The trust boundary is explicit in the UI: custom servers aren't moderator-reviewed, and the Add dialog says so.

## File manifest schema

Hosted at `files.fileManifestUrl`, regenerated by the operator on every patch (a generator script ships with the project):

```json
{
  "generated": "2026-06-12T00:00:00Z",
  "files": [
    { "path": "sunrunner_main.tre", "size": 48211344, "sha256": "9f86d08..." },
    { "path": "SWGEmu.exe",         "size": 6092800,  "sha256": "a1b2c3..." }
  ]
}
```

Each file is downloaded from `baseUrl + path`. Paths are relative to the server's game folder, forward slashes, no `..` segments (the launcher rejects any path that escapes the install folder). SHA-256 only:  MD5/CRC32 used by legacy launchers are not accepted.

## On-disk layout: one base, one folder per server

```
<install root>\
  base\                      pristine SWG 14.1 client files (shared, never modified)
  servers\
    sunrunner-home\          complete runnable game folder
    swg-legends\
  state\
    hashcache-<serverId>.json
```

Each server folder is a *complete* game install, which is what "executes with all the capability of their original launchers" requires:  but we avoid N full copies of a ~2 GB client. On install, every base-client file is **hard-linked** (NTFS `CreateHardLink`, zero extra disk) into the server folder, then the patcher overlays the server's files. A second server costs only its custom files, typically a few hundred MB.

Hard-link safety rule: the patcher must never open a linked file for writing (that would corrupt `base\` through the link). All downloads go to a `.lp-tmp` temp file, are hash-verified, then `File.Move(tmp, dest, overwrite: true)`:  which replaces the *link*, leaving base intact. This invariant is enforced in `PatchService` and must survive refactors.

## Patching protocol

Verify pass: walk the file manifest; a file needs download if missing, size differs, or SHA-256 differs. Hashing 2 GB on every check is slow, so a per-server hash cache maps `path → (size, lastWriteUtc, sha256)`; a file is only re-hashed when size or mtime changed. Download pass: up to 4 parallel downloads via `HttpClient`, temp-file + verify + atomic move per file, progress reported per-file and aggregate. A failed/corrupt download is retried twice, then surfaces as an error without aborting other files. "Verify files" in the UI is the same pass with the hash cache ignored.

## Login redirection and game settings

After patching, the launcher writes `login.cfg` in the server folder:

```
[ClientGame]
loginServerAddress0=login.example.org
loginServerPort0=44453
```

and ensures the client's root config (`swgemu.cfg`, or the executable name's `.cfg`) includes it (`.include "login.cfg"`). This is exactly how existing launchers retarget the stock client.

Settings editor: `options.cfg` in each server folder is INI-style; the launcher exposes the common knobs (resolution, windowed/fullscreen, shader level, sound) through a simple editor that round-trips unknown lines untouched. Per-server, because each server folder owns its own config.

Multi-instance: launching is `Process.Start` with `WorkingDirectory` = the server folder; pressing Play again starts a second client from the same folder (allowed unless the manifest says `allowMultiInstance: false`). Per-account profiles can come later.

## Accounts and launcher login

Two account models exist in the wild. SWGEmu-style servers take credentials at the in-game login screen:  manifest `auth.mode: "none"`, the launcher stays out of it. Legends-style servers authenticate in the launcher:  `auth.mode: "launcher"` adds an Account dialog (username, password, forgot-password and join-free links from the manifest). Credentials are stored per server, encrypted with Windows DPAPI under the current user, never in plain settings. If the manifest provides `loginUrl`, the launcher POSTs the credentials there on save (2xx = valid) for immediate feedback; if the server's client accepts credential arguments, `{username}`/`{password}` placeholders in `client.arguments` are substituted at launch. Play on an auth-required server with no saved account opens the dialog first.

## Trust and safety

The launcher downloads executables from third parties, so: registry inclusion is moderated (you approve manifest URLs); all URLs must be HTTPS; hash verification is mandatory; path traversal in manifests is rejected; the base folder is never writable by patch operations. Worth stating publicly in the submission guide: a server's listing can be pulled instantly by removing its registry entry. Future hardening: minisign/ed25519 signatures on file manifests, and a community-visible diff log on the registry repo (free if it's a GitHub repo:  PRs are the audit trail).

## What v1 deliberately skips

Account management (each server keeps its own signup), mod manager, self-update of the launcher itself (ship via GitHub Releases; add update check later), Linux/Wine support (the folder layout is Wine-friendly; revisit after v1), torrent/CDN distribution for first-install of the base client (operators already host full clients; the launcher can point at a zip later).
